# PAS-SCADA Health Monitor

Probes every component in the PAS-SCADA-Kafka-Bridge stack on a 30-second
cycle and emails ops the moment a component goes DOWN — and again the
moment it recovers.

Runs as a single Kubernetes deployment in the `pinkline` namespace.

**Built with FastAPI + uvicorn.** Endpoints exposed on port 8080:

| Path | What you get |
|---|---|
| `/` | Visual dashboard (HTML, auto-refreshes every 30s, color-coded UP/DOWN cards) |
| `/state` | JSON of every check's current state — for ad-hoc dashboards or curl |
| `/healthz`, `/readyz` | k8s liveness / readiness probes |
| `/docs` | Swagger UI for the above (auto-generated by FastAPI) |

## What it monitors

| Component        | Probe       | Endpoint |
|------------------|-------------|----------|
| PAS-SCADA Bridge | actuator    | `/actuator/health` (per-route + per-broker detail in alert) |
| Artemis OpenWire | tcp         | `61616` |
| Artemis Console  | http        | `/console/login` |
| Kafka            | tcp         | `9092` |
| Zookeeper        | tcp         | `2181` |
| RabbitMQ Mgmt    | http+basic  | `/api/healthchecks/node` |
| RabbitMQ AMQP    | tcp         | `5672` |
| RabbitMQ MQTT    | tcp         | `1883` |
| SCADA API        | http        | `/api/status` |
| SCADA API ↔ MQTT | json_field  | `/api/status` → `mqtt.connected == true` |

Add or remove checks by editing `k8s/10-configmap.yaml`.

## How alerting works

```
                  consecutive_fails >= fail_threshold
UP probe ─► OK     ─────►   DOWN  ──► email "DOWN: <name>"
                                       (no further alerts)

                  consecutive_passes >= recover_threshold
DOWN probe ─► RECOVER ──►  UP    ──► email "RECOVERED: <name> (down 2m 14s)"
```

* No alert spam: only fires on state transitions.
* State persisted to a 100Mi PVC, so a pod restart does not re-alert.
* Each DOWN email lists what *else* is currently DOWN — useful when a
  shared dependency takes multiple components with it.

## SMTP configuration

Non-sensitive (server, port, sender, recipients) → ConfigMap.
Credentials → Secret.

### Gmail

```bash
# 1. Enable 2-step verification on the sender Gmail account.
# 2. Create an App Password: https://myaccount.google.com/apppasswords
# 3. Use that 16-char password as SMTP_PASS.

kubectl -n pinkline create secret generic monitor-secret \
  --from-literal=SMTP_USER='alerts@yourcompany.com' \
  --from-literal=SMTP_PASS='abcd efgh ijkl mnop' \
  --from-literal=RABBITMQ_USER='thiru' \
  --from-literal=RABBITMQ_PASS='password' \
  --dry-run=client -o yaml | kubectl apply -f -
```

ConfigMap settings (edit `k8s/10-configmap.yaml`):

```yaml
SMTP_HOST: "smtp.gmail.com"
SMTP_PORT: "587"
SMTP_TLS:  "true"
SMTP_FROM: "alerts@yourcompany.com"
SMTP_TO:   "ops@yourcompany.com,oncall@yourcompany.com"
```

### Office 365

```yaml
SMTP_HOST: "smtp.office365.com"
SMTP_PORT: "587"
SMTP_TLS:  "true"
```

### Internal SMTP relay (no auth)

```yaml
SMTP_HOST: "smtp.internal.example.com"
SMTP_PORT: "25"
SMTP_TLS:  "false"
```

Leave `SMTP_USER` empty in the secret; the monitor will skip the LOGIN step.

## Deploy

```bash
# 1. Build and push the image (replace registry as needed)
docker build -t pinkline/pas-scada-monitor:latest ./monitor
docker push pinkline/pas-scada-monitor:latest

# 2. Set real credentials in k8s/20-secret.yaml (or apply a Secret directly)

# 3. Apply manifests in order
kubectl apply -f monitor/k8s/10-configmap.yaml
kubectl apply -f monitor/k8s/20-secret.yaml
kubectl apply -f monitor/k8s/30-pvc.yaml
kubectl apply -f monitor/k8s/40-deployment.yaml

# 4. Watch the logs
kubectl -n pinkline logs -f deploy/pas-scada-monitor
```

You should see one log line per check per cycle, like:

```
2026-04-27 14:30:01 [INFO] [OK]   pas-scada-bridge — UP
2026-04-27 14:30:01 [INFO] [OK]   kafka — connected kafka-service.pinkline.svc.cluster.local:9092
2026-04-27 14:30:01 [WARNING] [FAIL] rabbitmq-mgmt — ConnectionError: ...
```

## Verify alerting works (without breaking anything)

Trigger a synthetic DOWN by pointing one check at a non-existent host:

```bash
kubectl -n pinkline edit configmap monitor-config
# inside config.yaml, change one URL to http://does-not-exist:9999
kubectl -n pinkline rollout restart deploy/pas-scada-monitor
```

Wait `fail_threshold * interval_s` seconds (default: 3 × 30 = 90s) and
the DOWN email arrives. Revert the change → recovery email arrives.

## Tuning

| Symptom                                | Knob to turn |
|----------------------------------------|--------------|
| Too many alerts on transient blips     | Increase `fail_threshold` for that check |
| Slow detection of real outages         | Decrease `fail_threshold` or `interval_s` |
| Alerts during planned maintenance      | `kubectl scale deploy/pas-scada-monitor --replicas=0` for the window |
| Email-only is not enough               | Wrap a webhook on top — see "Adding alert channels" below |

## Adding new checks

Edit `k8s/10-configmap.yaml` and add to the `config.yaml` block:

```yaml
- name: my-new-thing
  type: http               # http | actuator | tcp | json_field
  url:  http://my-svc.namespace.svc.cluster.local:8080/health
  timeout_s: 5
  fail_threshold: 3
  recover_threshold: 2
```

Then:

```bash
kubectl apply -f monitor/k8s/10-configmap.yaml
kubectl -n pinkline rollout restart deploy/pas-scada-monitor
```

## Adding alert channels (Slack, PagerDuty)

Today the monitor sends SMTP only — keep it simple. Two patterns to extend:

1. **Mail-to-Slack** — most workspaces have an `email-to-channel` integration.
   Add the Slack-provided email to `SMTP_TO`. Zero code change.
2. **Webhook in `Alerter.send()`** — add ~15 lines: post JSON to a webhook
   URL alongside the SMTP send. Live in `monitor.py` under the `Alerter` class.

## Troubleshooting

**No email arriving, but log says EMAIL sent**
→ Check the sender's mail server logs / spam folder / sent items.
   For Gmail App Passwords, the sender must equal the account that owns
   the App Password.

**Log says `SMTP not configured`**
→ One of `SMTP_HOST`, `SMTP_FROM`, `SMTP_TO` is empty. Check the
   ConfigMap and Secret are applied: `kubectl -n pinkline get cm monitor-config -o yaml`.

**RabbitMQ probe always fails with HTTP 401**
→ `RABBITMQ_USER` / `RABBITMQ_PASS` in the Secret don't match the
   actual broker credentials. Test from another pod (RabbitMQ runs on
   the SCADA VM at `10.4.0.25`, mgmt on NodePort `31567`):
   `curl -u thiru:password http://10.4.0.25:31567/api/healthchecks/node`

**Pod CrashLoopBackOff with `config not found`**
→ ConfigMap not applied or wrong key. The deployment expects the
   ConfigMap to expose a `config.yaml` key — check
   `kubectl -n pinkline get cm monitor-config -o yaml | grep config.yaml`.

**Alert spam right after first deploy**
→ Expected: every check that's actually DOWN at startup will alert once,
   then go quiet. Adjust `fail_threshold` if your checks are flaky.
