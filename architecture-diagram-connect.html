Forward path — TMS → SCADA

TMS publish XML
 → Artemis topic TMS.PISInfo
 → tms-artemis-source (connect)
 → Kafka tms.raw  [on err: dlq.connect.tms-artemis-source]
 → tms/ Camel: XmlToJson + Encrypt
 → Kafka tms.scada.encrypted  [Camel err: Artemis DLQ.kafka-bridge]
 → tms-rabbitmq-sink (connect)
 → RabbitMQ amq.topic / key tms.scada.pas  [on err: dlq.connect.tms-rabbitmq-sink]
 → RMQ MQTT plugin → MQTT topic tms/scada/pas
 → SCADA API decrypts + displays

Reverse path — SCADA → TMS

SCADA publishes RSAE
 → MQTT scada/tms/alarms
 → RMQ queue scada.tms.alarms.queue
 → scada-rabbitmq-source (connect)
 → Kafka scada.tms.raw  [on err: dlq.connect.scada-rabbitmq-source]
 → tms/ Camel: Decrypt + JsonToXml(opt)
 → Kafka scada.tms.processed
 → scada-artemis-sink (connect)
 → Artemis topic SCADA.TMS.Alarms  [on err: dlq.connect.scada-artemis-sink]
 → TMS consumer reads

Cutover — flip toggles when Connect is verified

On day-1 deploy the toggles default to OFF — the Camel bridge runs exactly as it does today. Connect runs alongside but its output is unused. Verify, then flip: 

kubectl -n pinkline set env deploy/pas-scada-bridge \
  BRIDGE_INPUT_FROM_KAFKA=true \
  BRIDGE_PIPELINE=kafka \
  BRIDGE_REVERSE_KAFKA_ENABLED=true

After this:

  • Bridge stops consuming Artemis directly — reads tms.raw (from Connect's source) instead.
  • Bridge stops writing RabbitMQ/MQTT directly — writes only tms.scada.encrypted (Connect's sink delivers).
  • Bridge adds the reverse Kafka route (scada.tms.raw → decrypt → scada.tms.processed).

Rollback — set all three back to false / kafka,rabbitmq,mqtt and the bridge reverts to today's behavior.
