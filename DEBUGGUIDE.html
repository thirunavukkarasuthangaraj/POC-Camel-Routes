1. Understanding Log Levels — What You See in the Console

logging.level.org.apache.camel=INFO

# ── To see Kafka client internals ─────────────────────────
logging.level.org.apache.kafka=DEBUG

# ── To see Spring RabbitMQ internals ──────────────────────
logging.level.org.springframework.amqp=DEBUG

# ── To see ActiveMQ Artemis connection details ────────────
logging.level.org.apache.activemq=DEBUG
WHAT EACH LEVEL MEANS:
DEBUG
Shows every step — message body previews, processor internals, route transitions.
Use for: deep investigation
INFO
Shows main events — message received, published, forwarded.
Use for: normal monitoring
WARN
Shows retry attempts, unexpected-but-handled situations.
Use for: spotting problems early
ERROR
Shows failures — connection errors, encryption failures, message send failures.
Always investigate these

Step-by-Step: One XML message through the entire codebase
Trace a single ATRTimeTable XML from Artemis → SCADA
KafkaBridgeRoutes.java
from("activemq:topic:TMS.PISInfo?clientId=...&subscriptionDurable=true")

line 45–75

What to check in the XML→JSON conversion:
XML field Where it maps in POJO JSON output field
ATRTimeTable @dateTime ATRTimeTable.dateTime timestamp (envelope)
Tg @Id / @TripNo ATRTimeTable.Tg.id / .tripNo trainId / runNumber
Evts @As / @Ds ATRTimeTable.Evts.arrivalSecs / departureSecs arrivalTime / departureTime
Evts @F (flags) ATRTimeTable.Evts.flags Controls skip: 0x04=HISTORY, 0x20=PASS
Evts @PlatformTmsId ATRTimeTable.Evts.platformTmsId platformId = "PL" + value
Why platformPredictions is empty [] — how to debug
Most common JSON issue: empty array instead of train data
// In buildFromATRTimeTable() — add this to see which events are skipped:
for (ATRTimeTable.Evts evt : tg.evts) {

    // ─── DEBUG: print each event and why it might be skipped ───
    System.out.printf("  Evt: platformTmsId=%d  flags=0x%02X  As=%d  Ds=%d%n",
        evt.platformTmsId, evt.flags, evt.arrivalSecs, evt.departureSecs);

    if ((evt.flags & HISTORY_ITEM) != 0) {
        System.out.println("    SKIPPED — HISTORY_ITEM (flag 0x04)");
        continue;
    }
    if ((evt.flags & PASS_EVENT) != 0) {
        System.out.println("    SKIPPED — PASS_EVENT (flag 0x20)");
        continue;
    }
    if ((evt.flags & ARRIVAL_VALID) == 0 && (evt.flags & DEPARTURE_VALID) == 0) {
        System.out.println("    SKIPPED — neither ARRIVAL_VALID nor DEPARTURE_VALID");
        continue;
    }
    System.out.println("    → INCLUDED in output");
    // ... rest of loop
}
4. Debug Encryption
How to verify encryption is working correctly
processor/EncryptProcessor.java
Add to process() method after building payload:
// ─── TEMPORARY DEBUG — remove before production ────────────────
System.out.printf("[ENCRYPT] Input JSON length  : %d bytes%n", jsonBody.length());
System.out.printf("[ENCRYPT] IV (hex)           : %s%n", bytesToHex(iv, 12));
System.out.printf("[ENCRYPT] Encrypted+GCM len  : %d bytes%n", ciphertext.length);
System.out.printf("[ENCRYPT] Wire format total  : %d bytes (12 IV + %d cipher+tag)%n",
    payload.length, ciphertext.length);
System.out.printf("[ENCRYPT] First 6 bytes wire : %s...%n", bytesToHex(payload, 6));

// Helper method to add in EncryptProcessor:
private String bytesToHex(byte[] bytes, int limit) {
    StringBuilder sb = new StringBuilder();
    for (int i = 0; i < Math.min(limit, bytes.length); i++) {
        sb.append(String.format("%02X ", bytes[i]));
    }
    return sb.toString().trim();
}
Expected Debug Output
[ENCRYPT] Input JSON length  : 342 bytes
[ENCRYPT] IV (hex)           : A3 F2 C8 4D 11 9E 6B 2A 55 F0 D1 7C
[ENCRYPT] Encrypted+GCM len  : 374 bytes   (342 json + 16 GCM tag + 16 overhead)
[ENCRYPT] Wire format total  : 386 bytes (12 IV + 374 cipher+tag)
[ENCRYPT] First 6 bytes wire : A3 F2 C8 4D 11 9E ...
Key check: IV must be DIFFERENT every time you encrypt the same JSON — if IV repeats, security is broken. Run encrypt twice on same JSON and compare IV hex — they MUST differ.
SCADA_AES_KEY errors — how to diagnose
# Check if key is set (run in CMD before starting app):
echo %SCADA_AES_KEY%

# If blank → set it for local test:
set SCADA_AES_KEY=dGVzdGtleXRlc3RrZXl0ZXN0a2V5dGVzdGtleXRlc3Q=

# Check length (should be 44 chars for 32-byte base64):
echo %SCADA_AES_KEY% | wc -c   ← should print 45 (44 + newline)

# Verify it decodes to 32 bytes (PowerShell):
[System.Convert]::FromBase64String($env:SCADA_AES_KEY).Length   ← must print 32
Error: SCADA_AES_KEY not set
java.lang.IllegalStateException: SCADA_AES_KEY environment variable is not set!
  at EncryptProcessor.process(EncryptProcessor.java:42)
Fix: Set the env variable BEFORE running the app:
set SCADA_AES_KEY=dGVzdGtleXRlc3RrZXl0ZXN0a2V5dGVzdGtleXRlc3Q=
Error: Wrong key length
java.lang.IllegalStateException: SCADA_AES_KEY must decode to exactly 32 bytes (256-bit). Current: 16 bytes.
Fix: Generate a proper 32-byte key:
[Convert]::ToBase64String((1..32 | % { [byte](Get-Random -Max 256) }))

LAG > 0 means: messages arrived in Kafka but Route 2 (Kafka→RabbitMQ) hasn't consumed them yet.
Check: Is Route 2 running? Is RabbitMQ connection OK?
Kafka Connection Errors
Error: Cannot connect to Kafka
ERROR o.a.k.clients.NetworkClient : Connection refused to localhost:9092
WARN  o.a.camel.component.kafka   : Kafka producer is not yet connected
Cause: Kafka container not ready yet (takes ~20s after docker-compose up)
Fix: Wait 30 seconds. Check: docker ps — test-kafka must show "Up"
Check: docker logs test-kafka 2>&1 | tail -20 — look for "started (kafka.server.KafkaServer)"
Error: Topic not found
WARN  o.a.k.clients.producer : Topic tms.scada.encrypted not present
Cause: docker-compose.yml has KAFKA_AUTO_CREATE_TOPICS_ENABLE: "true" so topic auto-creates.
If it doesn't appear, Kafka may not be fully started yet.
Manual create: docker exec test-kafka kafka-topics --create --topic tms.scada.encrypted --bootstrap-server localhost:9092 --partitions 1 --replication-factor 1

7. All Common Errors — Cause & Fix
Error Reference Table
BUILD FAILURE — NoSuchMethodError
java.lang.NoSuchMethodError: 'javax.jms.ConnectionFactory...'
Cause: Old javax.jms import. Spring Boot 3 needs jakarta.jms
Fix: In any affected file change import javax.jms.* → import jakarta.jms.*
BUILD FAILURE — Cannot find symbol: camel-rabbitmq-starter
Could not find artifact org.apache.camel.springboot:camel-rabbitmq-starter:jar:4.4.0
Cause: camel-rabbitmq-starter doesn't exist in Camel 4.x
Fix: Use camel-spring-rabbitmq-starter + spring-boot-starter-amqp in pom.xml
TEST FAILURE — AEADBadTagException
javax.crypto.AEADBadTagException: Tag mismatch!
Cause: Decrypting with wrong key, OR message was tampered
Fix: Verify SCADA_AES_KEY is the SAME on both bridge and SCADA servers
Check: echo %SCADA_AES_KEY% on both servers — must be identical
RUNTIME — Empty platformPredictions []
{"messageType":"TMS_PAS_UPDATE","platformPredictions":[],...}   ← empty!
Cause 1: All Evts have HISTORY_ITEM flag (0x04) set → all skipped
Cause 2: ATRTimeTable.trains or .tg is null — Jackson didn't map XML correctly
Cause 3: XML element names don't match POJO field annotations
Debug: Add System.out.println(xml) at start of XmlToJsonProcessor.process()
RUNTIME — UnrecognizedPropertyException
com.fasterxml.jackson.databind.exc.UnrecognizedPropertyException: Unrecognized field "SomeField"
Cause: XML has a field not in the POJO model
Fix: Add @JsonIgnoreProperties(ignoreUnknown = true) to the POJO class
Or: Add the missing field to the POJO (ATRTimeTable.java, SingleArrival.java, etc.)
RUNTIME — Dead Letter Queue (DLQ) message
ERROR — message sent to dead-letter queue after 3 retries: <exception message>
Cause: A message failed in XmlToJsonProcessor or EncryptProcessor 3 times
Fix: Check the exception in the log line above the DLQ message
Check DLQ: Artemis console → Queues → DLQ.kafka-bridge → browse messages
RUNTIME — NullPointerException in buildFromATRTimeTable
java.lang.NullPointerException at XmlToJsonProcessor.buildFromATRTimeTable:line 95
Cause: timetable.trains or tg.evts is null (XML field missing)
Fix: Already guarded with if (timetable.trains != null) checks
Debug: Add System.out.println("trains: " + timetable.trains) before the loop
8. How to Add Debug Logging Properly (Using SLF4J)
The RIGHT way to add debug logs (already set up — just use them)
Each processor has a Logger already declared. Use log.debug() / log.info() / log.warn()
processor/XmlToJsonProcessor.java
processor/EncryptProcessor.java
// Logger already declared at top of each class:
private static final Logger log = LoggerFactory.getLogger(XmlToJsonProcessor.class);

// Use these in your code:
log.debug("Processing XML message, length: {}", xml.length());          ← only shows if DEBUG level
log.info("Converted to JSON, platforms found: {}", count);              ← always shows
log.warn("Unknown XML type, using minimal JSON. First 100 chars: {}", xml.substring(0, 100));
log.error("Encryption failed: {}", e.getMessage(), e);                  ← includes stack trace

// WRONG — never do this (security risk — exposes message body in logs):
log.debug("JSON body: " + json);        ← never log decrypted data in production
log.debug("AES key: " + keyB64);        ← NEVER log the key

Write a quick test for a new XML message type
Copy this pattern when you have a new XML to verify
// Add a new @Test method in XmlToJsonProcessorTest.java

@Test
void testMyNewXmlType() throws Exception {

    // 1. Paste the real XML here
    String xml = "<ATRTimeTable dateTime=\"20250115T103000\">"
               + "  <Trains><Tg Id=\"42\" TripNo=\"7\">"
               + "    <Evts As=\"600\" Ds=\"660\" PlatformTmsId=\"2201\" F=\"3\"/>"
               + "  </Tg></Trains></ATRTimeTable>";

    // 2. Run through processor
    XmlToJsonProcessor processor = new XmlToJsonProcessor();
    Exchange exchange = new DefaultExchange(new DefaultCamelContext());
    exchange.getIn().setBody(xml);
    processor.process(exchange);

    // 3. Get JSON output
    String json = exchange.getIn().getBody(String.class);
    System.out.println("JSON: " + json);

    // 4. Assert what you expect
    assertTrue(json.contains("TMS_PAS_UPDATE"));
    assertTrue(json.contains("PL2201"));        ← "PL" + PlatformTmsId
    assertTrue(json.contains("\"trainId\":\"42\""));
}
