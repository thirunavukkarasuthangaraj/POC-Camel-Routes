Overview — What This Topic Does
One Artemis Topic — Two XML Message Types
TMS publishes to this single topic for ALL real-time train movement events at platforms
TMS Server
→
RCS.E2K.TMS.TrafficReportClient
→
Bridge detects XML type
→
<SingleArrival> or <SingleDeparture>
SingleArrival
Train physically arrives at platform
→ status=1, arrivalTime=seconds now, departureTime=""
→ SCADA shows: ARRIVED, countdown stops
SingleDeparture
Train leaves the platform
→ status=2, arrivalTime=0, departureTime="14:33:00"
→ SCADA shows: DEPARTED, clears display
Why one topic for two types?
This is the TMS design. The topic is a general "traffic report" feed. The message type is identified by the XML root element name: <SingleArrival> or <SingleDeparture>.
The bridge detects which one it is in XmlToJsonProcessor.java line 61–66.

How arrivalTime is calculated — oTime → seconds from now
XmlToJsonProcessor.java
calcSecsFromNow() line ~265
// oTime from XML: "20260411T143000"
// Format:          yyyyMMdd'T'HHmmss

private long calcSecsFromNow(String oTime, long nowMs) {
    SimpleDateFormat fmt = new SimpleDateFormat("yyyyMMdd'T'HHmmss");
    fmt.setTimeZone(TimeZone.getTimeZone("UTC"));
    Date t = fmt.parse(oTime);
    return Math.max(0, (t.getTime() - nowMs) / 1000L);
    //           ↑ clamp to 0 — if oTime is in the past, result is 0
}

// Example:
// oTime   = "20260411T143000"  → 2026-04-11 14:30:00 UTC
// nowMs   = 2026-04-11 14:28:30 UTC
// diff    = 90 seconds
// result  = 90   → SCADA countdown shows "1 min 30 sec"

// If train arrived 10 seconds ago:
// oTime   = "20260411T143000"
// nowMs   = 2026-04-11 14:30:10 UTC
// diff    = -10 seconds
// result  = Math.max(0, -10) = 0   → SCADA shows "ARRIVED"

How departureTime is formatted — oTime → "HH:mm:ss"
XmlToJsonProcessor.java
formatTime() line ~280
private String formatTime(String oTime) {
    SimpleDateFormat in  = new SimpleDateFormat("yyyyMMdd'T'HHmmss");
    SimpleDateFormat out = new SimpleDateFormat("HH:mm:ss");
    in.setTimeZone(TimeZone.getTimeZone("UTC"));
    out.setTimeZone(TimeZone.getTimeZone("UTC"));
    return out.format(in.parse(oTime));
}

// Example:
// oTime  = "20260411T143300"
//           yyyy MM dd T HH mm ss
// result = "14:33:00"

// No clamping to 0 — departure time is always the actual scheduled time
// even if train departed in the past
POJO Classes — How Jackson Maps XML to Java
SingleArrival.java — Full Annotated Class
model/SingleArrival.java
@JacksonXmlRootElement(localName = "SingleArrival")   ← matches <SingleArrival> root tag
@JsonIgnoreProperties(ignoreUnknown = true)            ← extra XML fields are silently ignored
public class SingleArrival {

    @JacksonXmlProperty(localName = "Arr")             ← maps <Arr> child element
    public Arr arr;

    @JsonIgnoreProperties(ignoreUnknown = true)
    public static class Arr {

        @JacksonXmlProperty(localName = "train")
        public Train train;

        @JacksonXmlProperty(localName = "loc")
        public Loc loc;

        @JacksonXmlProperty(localName = "atimes")       ← note: "atimes" for arrival
        public Times atimes;
    }

    @JsonIgnoreProperties(ignoreUnknown = true)
    public static class Train {
        @JacksonXmlProperty(localName = "TTGUID")
        public String ttGuid;   ← V2: full GUID string

        @JacksonXmlProperty(localName = "id")
        public String id;       ← V1: numeric id as string
    }

    @JsonIgnoreProperties(ignoreUnknown = true)
    public static class Loc {
        @JacksonXmlProperty(localName = "tmsid")
        public int tmsId;       ← parsed as int: "2201" → 2201
    }

    @JsonIgnoreProperties(ignoreUnknown = true)
    public static class Times {
        @JacksonXmlProperty(localName = "oTime")
        public String oTime;    ← "20260411T143000" — kept as String, parsed later
    }
}
SingleDeparture.java — Comparison to SingleArrival
model/SingleDeparture.java
@JacksonXmlRootElement(localName = "SingleDeparture")  ← "SingleDeparture" not "SingleArrival"
@JsonIgnoreProperties(ignoreUnknown = true)
public class SingleDeparture {

    @JacksonXmlProperty(localName = "Dep")             ← "Dep" not "Arr"
    public Dep dep;

    public static class Dep {
        @JacksonXmlProperty(localName = "train")        ← same structure as Arrival
        public Train train;

        @JacksonXmlProperty(localName = "loc")
        public Loc loc;

        @JacksonXmlProperty(localName = "dtimes")       ← "dtimes" not "atimes" ← KEY DIFFERENCE
        public Times dtimes;
    }
    // Train, Loc, Times inner classes — identical to SingleArrival
    // dtimes.oTime = departure time, formatted as "HH:mm:ss" (not converted to seconds)
}
Why separate classes? Even though the fields look similar, Arrival has <Arr><atimes> and Departure has <Dep><dtimes>. Different XML element names — different Jackson annotations — separate POJO classes.
Detection Logic — How the Bridge Knows Which Type It Received
XmlToJsonProcessor — Type Detection Code
processor/XmlToJsonProcessor.java
process() line 55–70
@Override
public void process(Exchange exchange) throws Exception {

    String xml    = exchange.getIn().getBody(String.class);
    String schema = exchange.getIn().getHeader("rcsschema", String.class);
    // schema = e.g. "RCS.E2K.TMS.TrafficReportClient.SingleArrival.V2"
    //           or  "RCS.E2K.TMS.TrafficReportClient.SingleDeparture.V2"
    // (set by Artemis as JMS message property — not used for routing, only for logging)

    // Type detection — fast string check on root XML element
    if (xml.contains("<ATRTimeTable")) {
        ATRTimeTable timetable = xmlMapper.readValue(xml, ATRTimeTable.class);
        json = buildFromATRTimeTable(timetable, schema);

    } else if (xml.contains("<SingleArrival")) {            ← checks for this string anywhere in XML
        SingleArrival arrival = xmlMapper.readValue(xml, SingleArrival.class);
        json = buildFromSingleArrival(arrival);

    } else if (xml.contains("<SingleDeparture")) {          ← checks for this string
        SingleDeparture departure = xmlMapper.readValue(xml, SingleDeparture.class);
        json = buildFromSingleDeparture(departure);

    } else if (xml.contains("<routeinfo")) {
        RouteInfo routeInfo = xmlMapper.readValue(xml, RouteInfo.class);
        json = buildFromRouteInfo(routeInfo);

    } else {
        log.warn("Unknown XML message type — building minimal JSON envelope");
        json = buildMinimalJson();
    }
}
Detection order matters. SingleArrival is checked before SingleDeparture. If TMS ever sends a message with both strings (e.g. a wrapped envelope), SingleArrival wins. In practice, each message has only one root element.

private String resolveTrainId(String ttGuid, String id) {
    if (ttGuid != null && !ttGuid.isBlank()) return ttGuid;  ← V2: GUID preferred
    if (id     != null && !id.isBlank())     return id;      ← V1: numeric id fallback
    return "0";                                                ← last resort
}

// Called with:
// msg.arr.train.ttGuid  ← from <TTGUID> element
// msg.arr.train.id      ← from <id> element

// Result examples:
// V2 XML:  ttGuid="dc-occ.eclrt-train-6250-guid", id="6250"  → "dc-occ.eclrt-train-6250-guid"
// V1 XML:  ttGuid=null,                            id="6250"  → "6250"
// Bad XML: ttGuid=null,                            id=null    → "0"

Status Values — What SCADA Does With Each
status field — train state at platform
status Meaning Set by SCADA display action
0 Predicted (not yet arrived) ATRTimeTable messages Show countdown timer "arriving in X min"
1 At platform (arrived) SingleArrival messages Stop countdown, show "ARRIVED" / "NOW BOARDING"
2 Departed SingleDeparture messages Clear the platform display, show departure time
serviceState field
serviceState Meaning SCADA display action
1 Will stop at platform Normal display — show arrival/departure info
2 Not stopping (pass-through) Show "NO STOP" or suppress display
For SingleArrival and SingleDeparture, serviceState is hardcoded to 1 — if a train passed through without stopping, TMS would not send a SingleArrival/SingleDeparture at all. The pass-through filtering happens upstream in TMS.
Real Message Timeline — Arrival Then Departure Sequence
What happens on topic RCS.E2K.TMS.TrafficReportClient for one train stop
Full sequence: prediction → arrival → departure
T-5 min: ATRTimeTable published on TMS.PISInfo
Bridge gets ATRTimeTable → builds prediction → arrivalTime=300 (5 min) → SCADA shows countdown
T+0: SingleArrival on RCS.E2K.TMS.TrafficReportClient
oTime = "20260411T143000" → bridge calculates: past time → arrivalTime=0
JSON: status=1, arrivalTime=0, departureTime=""
SCADA: stops countdown → shows "ARRIVED"
T+3 min: SingleDeparture on RCS.E2K.TMS.TrafficReportClient
oTime = "20260411T143300" → bridge formats: "14:33:00"
JSON: status=2, arrivalTime=0, departureTime="14:33:00"
SCADA: shows "DEPARTED 14:33" → clears platform display
